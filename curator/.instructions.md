# Curator VS Code Extension — Agent Instructions

## Purpose
Expose the Padagaskar wiki (WIKI_PAGE sources) as a virtual filesystem in VS Code, allowing users to browse, read, and (eventually) edit wiki pages as files. Authentication is handled securely using the SecretStorage API and a loopback OAuth flow.

## Key Behaviors
- Only WIKI_PAGE sources are shown as files/folders.
- All API requests use the JWT stored in SecretStorage (never in files or settings).
- The extension provides a `Curator: Sign In to Wiki` command for seamless authentication.
- Uses the built-in fetch API (Node.js 18+) for all HTTP requests.
- All extension logic requiring authentication or workspace context must use the context object passed to `activate`.
- Never store secrets or tokens in globalState, files, or logs.

## Best Practices
- Always check for a valid token in SecretStorage before making API requests.
- If the token is missing or expired, prompt the user to sign in.
- Handle API/network errors gracefully and show user-friendly messages.
- Use the VS Code activity bar and custom views for navigation.
- Keep all authentication and API logic in dedicated modules (e.g., `auth.ts`, `api.ts`).

## Security
- All secrets must be stored and retrieved using `context.secrets`.
- Never expose tokens in logs, errors, or UI.
- Use HTTPS for all API endpoints.

# Developer Notes
- All extension entry logic must be inside the `activate(context)` function.
- Pass the `context` object to any class or function that needs access to secrets or workspace APIs.
- Use the built-in fetch API (no need for node-fetch in Node.js 18+).
- For new features, follow the established authentication and API patterns.
